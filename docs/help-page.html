Why CloudScale Free Backup and Restore?

Most WordPress backup plugins lure you in with a free tier — and then lock the restore button behind a paywall. UpdraftPlus Premium costs $70/year. BackupBuddy starts at $80/year. Duplicator Pro is $69/year. All of them charge you at exactly the moment you are most desperate: when your site is broken and you need to get it back.

CloudScale is free. Not "free with paid restore." Free. Scheduled or manual backups, one-click restore, maintenance mode, S3 cloud sync, and Google Drive sync — all included, no subscription required, source code on GitHub.

Your backups are stored on your server and can be automatically synced to Amazon S3 or Google Drive so a server failure never means data loss. Download any backup to your computer at any time.

The Backup List shows every backup stored on your server. Each entry is a single .zip file containing your full site snapshot — database SQL dump, wp-content/uploads/, all active plugins, and the active theme.

  • Date / time — exact timestamp of when the backup completed, displayed in your WordPress timezone.
  • Size — uncompressed zip file size. A typical WordPress site with a moderate media library produces backups of 200 MB–2 GB. Monitor this to avoid filling your disk; the Retention setting automatically trims oldest backups.
  • Type — Manual (triggered by you via the admin panel) or Scheduled (automatic, triggered by WordPress cron or system cron).
  • Download — streams the zip directly to your browser. Always download at least one backup offsite — a server failure would destroy both your site and any backups stored on the same server.
  • Restore — starts the restore process for that backup. See the Restore & Maintenance section for the full restore sequence and what to expect.
  • Delete — permanently removes the zip from the server. Cannot be undone.

Storage location: backups are stored in wp-content/cloudscale-backups/. This directory is created automatically on first backup. To move backups to a different partition or mount point, symlink this directory to your preferred location before creating the first backup.

Create a Backup

Click Create Backup Now to immediately create a full site backup. The backup runs synchronously in the browser — keep the tab open until the success message appears. For large sites (>1 GB uploads), increase max_execution_time in php.ini to at least 300 seconds to avoid timeout.

What is included in every backup:

  • Complete WordPress database exported as a .sql dump — includes all tables with the configured $table_prefix, covering posts, pages, users, settings, plugin data, and custom tables.
  • wp-content/uploads/ — your full media library including all image sizes generated by WordPress.
  • wp-content/plugins/ — all installed plugins (not just active ones).
  • Active theme directory under wp-content/themes/.

What is NOT included: wp-config.php, .htaccess, WordPress core files (wp-admin/, wp-includes/). These are excluded intentionally — restoring core files from a backup can conflict with subsequent WordPress upgrades, and wp-config.php contains environment-specific credentials that should not be moved between servers.

Scheduled backups — configure automatic backups to run daily, twice-daily, or weekly using WordPress cron. For production sites, use a system cron entry instead of WP-Cron to guarantee the schedule runs regardless of traffic: * * * * * curl -s https://yoursite.com/wp-cron.php?doing_wp_cron > /dev/null

Retention — set the maximum number of backups to keep on-server. When a new backup is created and the limit is exceeded, the oldest backup is deleted automatically. Recommended: keep 3–7 backups on-server and download one offsite weekly.

Restore & Maintenance

Restore replaces your live site with the contents of the selected backup zip. The plugin activates Maintenance Mode during the entire process, serving a "Down for maintenance" HTTP 503 response to all frontend visitors. The wp-admin remains accessible to administrators.

Full restore sequence:

  1. Maintenance mode is activated — wp-content/.maintenance is written to the filesystem.
  2. The backup zip is extracted to a temporary directory inside wp-content/.
  3. The current database is replaced: existing tables matching the backup's table prefix are dropped and recreated from the .sql dump. Tables with different prefixes are left untouched.
  4. The uploads/ directory is replaced with the backup's media files.
  5. The plugins/ and active theme directories are replaced.
  6. Maintenance mode is deactivated — the .maintenance file is deleted.
  7. OPcache is flushed to ensure PHP loads the restored files immediately.

Important: A restore is irreversible. Any database changes, new posts, uploaded media, or plugin updates made after the backup date are permanently lost. Always create a fresh backup immediately before restoring if you want a recovery point.

If a restore fails mid-way — maintenance mode remains active. You can manually delete wp-content/.maintenance via SFTP or SSH to bring the site back online, then investigate the error log at wp-content/debug.log.

Restoring to a different server or domain — after restoring to a new server, run a search-replace on the database to update the old domain to the new one: wp search-replace 'https://oldsite.com' 'https://newsite.com' --all-tables
